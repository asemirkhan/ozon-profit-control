# ETL Blueprint v1 — Ozon Profit Control

## 0. Зачем этот документ
Этот документ описывает, **как именно** мы будем забирать данные из Ozon, сохранять их, превращать в таблицы и строить витрины для Power BI.
Он нужен, чтобы:
- не “изобретать заново” на каждом клиенте,
- не потеряться в данных,
- уменьшить количество ошибок,
- быстро получать стабильный результат.

## 1. Общая схема (что куда течёт)
**Ozon API → Raw JSON → Staging tables → Mart tables → Power BI**

### 1.1 Что такое Raw JSON
Raw JSON — это “сырые” ответы API, сохранённые как есть.
Мы храним их всегда, потому что:
- можно перепроиграть обработку, не дёргая API повторно,
- легче дебажить (видно, что реально пришло),
- если API поменяется — у нас останется история.

### 1.2 Что такое Staging tables
Staging — это таблицы, которые **максимально похожи на API по смыслу**, но уже табличные:
- одна строка = одна сущность (отправление, позиция, транзакция и т.д.)
- нормальные колонки
- удобно джойнить SQL’ом

### 1.3 Что такое Mart tables (витрины)
Mart — это готовые “витрины” для BI:
- быстрые,
- стабильные,
- без тяжёлых джойнов в Power BI.

Power BI должен читать **только mart_*** (а не сырьё).

## 2. Какие данные забираем из Ozon (в v1)
В v1 мы берём 5 источников:
1) Postings (отправления/заказы) — что продали и в каких статусах
2) Postings items (состав отправлений) — какие товары и сколько
3) Finance transactions — начисления/удержания/услуги (почему денег “к выплате” меньше)
4) Products list — справочник товаров (offer_id ↔ sku ↔ product_id)
5) Ads stats — расходы на рекламу (минимум для прибыли после рекламы)
(Остатки — опционально, если источник доступен)

Точные эндпойнты и поля лежат в документе `docs/api_endpoints_ozon_v1.md`.

## 3. Пайплайн: шаги выполнения (в правильном порядке)

### Step 1 — Ingest (забор данных из API)
Что делаем:
- запрашиваем данные по каждому источнику за нужный период
- учитываем лимиты по периоду (например, транзакции часто берутся месяцами)
- для больших периодов делим запросы на куски

Что сохраняем:
- сохраняем raw JSON ответ **каждого запроса** в файловое хранилище

Структура папок raw (пример):
- `data/raw/<client_slug>/postings/YYYY-MM-DD/*.json`
- `data/raw/<client_slug>/transactions/YYYY-MM-DD/*.json`
- `data/raw/<client_slug>/products/YYYY-MM-DD/*.json`
- `data/raw/<client_slug>/ads/YYYY-MM-DD/*.json`

Почему так:
- понятно, где что лежит,
- удобно пересобирать за конкретный день/период,
- можно хранить историю.

### Step 2 — Normalize (raw → staging)
Что делаем:
- читаем raw JSON
- “расплющиваем” вложенные массивы (например products[], services[], items[])
- приводим типы (даты, числа)
- пишем результат в staging таблицы в Postgres

Какие staging таблицы строим:
- `dim_product`
- `stg_posting`
- `stg_posting_item`
- `stg_posting_item_financial`
- `stg_transaction`
- `stg_transaction_item`
- `stg_transaction_service`
- `fact_ads_spend`
- `fact_cogs` (не из API — загружается из таблицы клиента)

Важно:
- staging таблицы не должны “умничать” — они отражают данные как есть, но в табличной форме.

### Step 3 — Data Quality Checks (обязательная проверка данных)
Это проверки, которые предотвращают “тихий брак”, когда отчёт строится, но цифры неправильные.

Минимальный набор проверок v1:
1) В `stg_posting` нет дублей `posting_number`
2) В `stg_transaction` нет дублей `operation_id`
3) В `stg_posting_item` нет строк с `quantity <= 0`
4) Покрытие себестоимости:
   - минимум 80% продаж (units_sold или payout) должны иметь найденный `unit_cogs`
5) Сопоставление товаров:
   - доля строк, где есть `offer_id` и/или `sku` и они маппятся на `dim_product`, должна быть высокой
6) Базовая сверка денег:
   - суммарный `payout` по postings за период должен быть “похож” на финансовые данные (с допуском и пониманием лагов)

Результат:
- записываем таблицу `dq_report` (или файл) со статусами `OK/WARN/ERROR`
- если `ERROR` — останавливаем сборку витрин и фиксируем причину.

### Step 4 — Build Marts (staging → mart)
Мы строим витрины (mart) — подготовленные таблицы для Power BI.

Витрины v1:
1) `mart_pnl_daily_by_product`
2) `mart_pnl_30d_by_product`
3) `mart_leakage_summary_30d`
4) `mart_ads_daily` (если реклама подключена)
5) `mart_inventory_snapshot` (если остатки подключены)
6) `mart_executive_summary_30d`

Примечание:
- точная логика и состав полей витрин описаны в `docs/data_contract_v1.md`.

### Step 5 — Publish to BI (Power BI)
Power BI подключается **только к mart_ таблицам**.
Это принцип:
- меньше ошибок,
- быстрее отчёты,
- проще поддержка.

## 4. Расписание обновлений (v1)
Цель: “ежедневное обновление без ручного труда”.

Рекомендуемое расписание:
- Postings + Transactions: ежедневно ночью
- Products list: 1 раз в сутки (или раз в неделю, если редко меняется)
- Ads stats: ежедневно (если используем)
- Marts: сразу после обновления staging
- DQ checks: перед построением marts

## 5. Поддержка нескольких клиентов (как не утонуть)
Каждый клиент имеет:
- `client_slug` (короткое имя, например `client_001`)
- свой конфиг (файл), где лежат:
  - API ключи/идентификаторы (НЕ в git)
  - период обновления
  - особенности (FBS/FBO, есть ли реклама, есть ли остатки)
- своё пространство в базе:
  - вариант A: отдельная схема (schema) в Postgres на клиента
  - вариант B: общие таблицы + колонка `client_id` (в v1 проще отдельные схемы)

Мы начинаем с варианта A (отдельная схема), потому что:
- проще изолировать клиентов,
- меньше рисков смешать данные.

## 6. Логи, ошибки, воспроизводимость (чтобы проект был “промышленным”)
Мы делаем так:
- каждый запуск ETL получает `run_id`
- все шаги пишут логи с `run_id` (ingest/normalize/dq/marts)
- raw JSON всегда сохраняется
- staging/mart таблицы имеют `raw_updated_at` или `loaded_at` (когда данные загружены)

Если клиент говорит “вчера цифры были другие”:
- мы можем показать, какие данные пришли из API,
- когда они были загружены,
- что изменилось.

## 7. Definition of Done для ETL v1
ETL v1 считается готовым, если:
- есть команды/скрипты, которые собирают staging таблицы
- строятся все обязательные витрины (pnl daily, pnl 30d, leakage summary, executive summary)
- есть DQ отчёт и он реально останавливает сборку при критичных ошибках
- Power BI подключается к Postgres и отчёт работает быстро на mart таблицах
